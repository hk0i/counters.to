# Engineering Design Document (EDD)

## Project: Counters.to Web Frontend (V1)

**Status:** Draft
**Architecture:** Static Vue 3 SPA / Offline-First PWA, consuming the flat-file-compiled REST API from `counters-data-core`

## 1. Objective & Scope

This specification defines the frontend that consumes the static JSON API produced by `packages/counters-data-core` (see `docs/counters.to backend EDD.md`). It covers stack choice, directory layout, data-fetching and offline strategy, routing/page map, state management, build/deploy to GitHub Pages, and testing. Visual/brand design is explicitly out of scope for V1 — see section 9.

This package is licensed **GPLv3**, separately from `counters-data-core` (MIT). See section 10.

## 2. Stack

- **Vue 3** (Composition API, `<script setup>`), **TypeScript**, **Vite**
- **vue-router** for client-side routing
- **vite-plugin-pwa** (Workbox under the hood) for service worker + manifest generation
- No Pinia, no Nuxt. Reasoning below.

### Why not Nuxt

Nuxt's SSR/hybrid-rendering model solves problems this project doesn't have — there's no server, no per-request personalization, and the data is fully static at build time. The main thing Nuxt would buy us is pre-rendered HTML per hero page for SEO. That's a real tradeoff (a pure client-side SPA means hero detail pages have no meaningful content in their initial HTML for crawlers that don't execute JS), but it's an acceptable V1 cost given the backend EDD's own "pragmatic V1, migrate later" pattern. If hero-page SEO becomes a priority, revisit with `vite-ssg` (adds static pre-rendering to a Vite+Vue app without Nuxt's full framework) rather than adopting Nuxt wholesale.

### Why not Pinia

The app's shared state is small enough (selected enemy team slots, theme, network/offline status) to live in a couple of composables using Vue's built-in `reactive`/`ref`. Add Pinia later if state complexity grows past a few composables — it's a low-cost migration, not worth the dependency up front.

### Why this pairs well with the existing backend design

The backend EDD's `packages/counters-web/public/api/v1/` output location isn't incidental — Vite copies everything in a project's `public/` directory into `dist/` verbatim at build time. Because `compile.ts` already writes into *this* package's `public/` dir, `npm run build` here ships the compiled API and the built SPA as one static artifact with zero extra plumbing. This is a load-bearing reason to stay on a Vite-based toolchain rather than something with a different static-asset convention.

## 4. Data Layer

### 4.1 New backend dependency: `matchups/index.json`

The team-builder view (section 6) needs pairwise matchup data for up to 5 heroes at once. Fetching 5 separate `heroes/:id/index.json` deep resources per lookup works but is wasteful. This spec assumes `counters-data-core` adds one more static endpoint:

```
GET /api/v1/matchups
Physical path: public/api/v1/matchups/index.json
```

A flat structure covering every ordered pair where a relationship exists (both directions checked independently, per the backend's non-reciprocal design — see `CLAUDE.md`):

```json
{
  "data": [
    { "heroId": "sigma", "threatFrom": "zarya", "matchedTraits": ["beam"] },
    { "heroId": "sigma", "advantageOver": "dva", "matchedTraits": ["projectile"] }
  ]
}
```

~2,600 entries at 51 heroes, comfortably one small fetch. **This endpoint doesn't exist yet** — it's a dependency on the data-core package, not something this frontend spec can build alone. Flag it back as a small follow-up task there before starting `TeamBuilderView`.

### 4.2 Fetching & caching contract

- `useHeroes()` fetches `/api/v1/heroes/index.json` once, caches in a module-level `ref` for the session (no need to refetch on every route change).
- `useHero(id)` fetches `/api/v1/heroes/:id/index.json` lazily per hero-detail visit, cached the same way.
- `useMatchups()` fetches `/api/v1/matchups/index.json` once, same pattern.
- All fetches are plain `fetch()` — no need for a data-fetching library (TanStack Query, SWR, etc.) at this scale. Revisit if request complexity grows (retries, pagination, mutations — none of which apply to a read-only static API).

### 4.3 Offline strategy (`vite-plugin-pwa`)

- **Precache** (install-time, via Workbox `globPatterns`): the app shell — built JS/CSS/HTML and `heroes/index.json` (small, needed for the home view to work offline immediately).
- **Runtime cache** (`StaleWhileRevalidate`): everything else under `/api/v1/**` — serves instantly from cache, refreshes in the background. Appropriate because this data only changes on redeploy (hero balance patches), not in real time; `CacheFirst` would risk staleness after a deploy, `NetworkFirst` would defeat the offline-first goal.
- `OfflineBanner.vue` listens to `useOnlineStatus()` (wraps `navigator.onLine` + `online`/`offline` events) and shows a small persistent notice when offline — data is still fully usable, just not guaranteed fresh.

## 5. Routing / Page Map

| Route | View | Purpose |
|---|---|---|
| `/` | `HomeView` | Hero grid, filterable by role, searchable by name |
| `/heroes/:id` | `HeroDetailView` | Playbook, tactical caveats, threats list, advantages list |
| `/team-builder` | `TeamBuilderView` | Pick an enemy team (role-locked, up to 1 tank / 2 damage / 2 support), get ranked counter suggestions computed client-side from `useMatchups()` |
| `/about` | `AboutView` | Project description, license summary, Blizzard IP attribution (section 10) |

All client-side (`vue-router` in `history` mode). Since this is a fully static SPA with no server to handle deep-link fallback, GitHub Pages needs a `404.html` that redirects to `index.html` (the standard SPA-on-GitHub-Pages workaround) — add this alongside the `CNAME` file in `public/`.

## 6. Team-Builder Aggregation Logic

Client-side, no backend involvement beyond the `matchups` fetch:

1. User fills up to 5 enemy team slots (role-locked to mirror the 1-2-2 structure).
2. For each candidate hero not on the enemy team, sum matched-trait counts where the candidate's `advantageOver` hits an enemy slot, minus counts where the candidate would be threatened by an enemy slot.
3. Sort candidates by that score, descending, grouped by role for the suggestion panel.

This is intentionally simple (a weighted sum, not a real matchmaking algorithm) — good enough for V1 and easy to reason about. Note in `AboutView` or nearby copy that suggestions are heuristic, not a guarantee.

## 7. Component Responsibilities (high level)

- **HeroCard** — role icon, name, click-through to detail. No fetching, pure props.
- **RoleFilterBar** — emits a role filter; `HomeView` owns the filtered list.
- **TeamSlotPicker** — 5 role-locked slots, hero picker per slot (reuses `HeroCard` in a compact mode).
- **CounterSuggestionsPanel** — renders the ranked output of section 6's aggregation.
- **MatchupBadge** — small "threat" / "advantage" chip with matched-trait tooltip, reused on `HeroDetailView` and `CounterSuggestionsPanel`.
- **OfflineBanner** — global, mounted once in `App.vue`.

Prop-level contracts and full component specs are deferred to implementation — this list exists so the view/component boundary is agreed before code exists, not to fully replace normal PR review.

## 8. Build & Deploy (GitHub Pages)

### Custom domain assumption

The project is named `counters.to` — this spec assumes a **custom domain** deploy (`public/CNAME` containing `counters.to`), not a `username.github.io/counters.to/` project-pages subpath. That means `vite.config.ts` should set `base: '/'` (the default), not a repo-name subpath. **Confirm this before first deploy** — if you end up on a subpath instead, `base` needs to change and every absolute `/api/v1/...` fetch call needs to become relative or environment-aware.

### CI pipeline (GitHub Actions)

This closes the "no CI/deploy trigger" gap flagged in the backend EDD review. On push to `main`:

1. Checkout, setup Node, `npm ci` (workspace root).
2. `npm run compile --workspace=counters-data-core` — regenerate `public/api/v1/**` from the current hero `.md` source, so the deployed site always reflects the latest committed hero data rather than a possibly-stale committed JSON snapshot.
3. `npm run build --workspace=counters-web` — Vite build; `public/` (including the freshly regenerated API JSON) copies into `dist/` automatically.
4. Upload `dist/` as a Pages artifact, deploy via `actions/deploy-pages`.

Whether the compiled `public/api/v1/**` JSON stays committed to git after CI does this for every deploy is a separate question worth revisiting — it's currently committed so the repo is inspectable/diffable without running the pipeline, but CI regenerating it on every deploy makes the committed copy closer to a convenience snapshot than a build artifact.

## 11. Testing Strategy

Absent from the backend EDD's review, worth not repeating here:

- **Vitest** + **Vue Testing Library** for component/composable tests — particularly `useTeamComp` and the section 6 aggregation logic, since that's the one piece of real client-side logic in an otherwise mostly-presentational app.
- **Playwright**, scoped narrowly: one smoke test that the app loads and shows heroes, one that offline mode (service worker cache) actually serves the home view with the network disabled. Full e2e coverage isn't proportionate to this project's size — expand only if the app's surface area grows.
